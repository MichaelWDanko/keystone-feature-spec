How to use Keystone

Keystone stores feature requirements in Markdown files within your repository. This guide explains how to add those files, what each file contains, and how to use them during implementation.

Open a worked example Read agent guidance →

Use it in a project

Set up a project in three steps.

  1. 1
    Create feature-spec/

    Put one Markdown file per feature area directly in this directory. Keep it next to the implementation, not in a separate system.

  2. 2
    Write the intended behavior

    Use observable requirements with clear words such as MUST and MUST NOT. Avoid encoding the current implementation.

  3. 3
    Make it part of the change

    Before changing behavior, read the relevant specification. Update it only when the feature requirements change too.

Inside a spec file

What a specification file contains.

Each file starts with its name and purpose. The sections below it record feature state, requirements, evidence, and related context.

File header

# Settings.Accounts

Canonical name and purpose
The H1 exactly matches the filename. The opening paragraph defines the specification’s scope.

Optional and recommended sections

## Status

Feature state
Required, Implemented, Deferred, Deprecated, or Retired.

## Requirements

Feature requirements
Observable behavior and durable constraints.

## Verification

Evidence
What tests or review should prove.

## Exceptions

Narrow differences
The parent source, exception, and rationale.

## Related specifications

Other context
Non-parent specifications to inspect before a change.

Finding the right context

Use filenames to find the right context.

Use dots instead of nested folders. A more specific document inherits every requirement from its existing parents.

feature-spec/
  Settings.md
  Settings.Accounts.md
  Settings.Accounts.Removal.md

For a change to account removal, read the files from broadest to most specific. Together they define what the feature must do.

  1. Settings.md
  2. Settings.Accounts.md
  3. Settings.Accounts.Removal.md

A child can add or strengthen a requirement. It cannot silently weaken a parent; when behavior must differ, document a narrow exception.

When changing code

Read the specification before implementation.

01

Identify the most specific affected specification. Start with filenames, then read only its parent chain and related specifications.

02

Preserve its effective requirements and verification. Missing code or tests is not evidence that a documented capability is obsolete.

03

Make feature changes explicit. If a requirement no longer holds, update the specification deliberately and record the reason.
