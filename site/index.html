Why Getting started Example Specification

Keep essential feature requirements beside the code.

Keystone captures the behavior a project intends to preserve. It does not need to describe everything. Tests alone cannot protect a defined feature: an agent can remove the behavior and its tests while leaving the remaining test suite green. Keystone keeps selected requirements outside the implementation so people and coding agents can review what must remain before changing code.

Tests answer Does the code behave as expected?
Keystone answers Does an agent's changes preserve the intended features?

KEYSTONE.md explains the project and guides agents.

This root-level file explains what is being built, distinguishes relevant surfaces, and maps them to feature namespaces. Its project-specific context is descriptive. Essential behavior belongs in normative requirements under feature-spec/.

README.md remains the human-facing overview, while AGENTS.md contains repository-wide working guidance.

# Keystone agent guidance

This project uses Keystone to keep essential feature requirements
separate from their implementation. Treat the Markdown documents
in `feature-spec/` as behavior the project intends to preserve.

## Feature context

This example project runs Leash, a dog-walking app. Dog owners
use the app to manage dogs and request walks. Dog walkers use a
separate workspace to manage availability and assigned walks.

- `Leash.Settings.*` describes shared app settings and safety controls.
- `Leash.Owners.*` describes the dog-owner experience.
- `Leash.Walkers.*` describes the dog-walker experience.

## Before changing a Keystone feature

Before changing a documented feature, an agent MUST:

1. identify the most specific affected specification;
2. read its namespace parents from least to most specific;
3. read the target and its related specifications;
4. identify inherited requirements and exceptions;
5. make implementation and tests conform; and
6. report conflicts instead of weakening a requirement.

When an agent adds behavior that is not covered by an existing specification,
it asks whether the feature belongs in Keystone's scope. If yes, it proposes
the smallest suitable namespace and requirements for confirmation before
writing the specification. If no, the behavior remains outside the documented
collection by design.

## Navigate `feature-spec/` efficiently

Do not load every file in `feature-spec/`. Start with filenames,
derive the parent chain from the dot-separated name, and open
only that chain and relevant related specifications.

Describe the features that must stay.

Keystone does not require you to document every feature. For each feature of your application, create a Markdown file named after it. Describe what it must do, not how the code is built.

Put shared requirements at the namespace root. A single-product project can use a root such as Leash.md for product-wide guarantees inherited by Leash.Owners.md and Leash.Walkers.md. A monorepo can instead use separate roots such as Desktop.md, Server.md, and Shared.md.

Need to describe something more specific? Add another part to the filename, separated by a dot. For example, Leash.Walkers.Availability.md includes the requirements from Leash.Walkers.md because its name starts with that name. The new file only needs the added requirements for showing walker availability.

# Leash.Walkers

Defines requirements for dog walkers who use the fictional Leash dog-walking app.

## Requirements

- A walker profile MUST identify the walker and their service area.
- An unavailable walker MUST NOT be assigned a new walk.
- Walker views MUST identify the dog, owner, location, and care instructions.
- Updating one walker profile MUST NOT change another walker profile.
